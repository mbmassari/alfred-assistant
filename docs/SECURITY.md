# Security — Assistente Alfred

## Authentication

- **Single bearer token** stored as file (`secrets/gateway_auth_token`)
- Token is 64 characters (hex), auto-generated by `init-secrets.sh`
- All API endpoints (except `/health` and `/auth/verify`) require the token
- Token rotation: `make rotate-token`

## Network Isolation

### Docker Networks
- 3 separate networks isolate components
- Backend and MCP networks are `internal: true` (no external access)
- Only the gateway and Portainer are exposed to the host

### Nanobot Egress Firewall
- iptables allowlist in `entrypoint.sh`
- Only whitelisted domains are reachable (OpenRouter, Gmail, Telegram, etc.)
- Default policy: DROP all other outbound traffic
- Container has `NET_ADMIN` capability for iptables configuration

## Secret Management

- Secret values stored as files with `0600` permissions
- Never stored in database (only metadata/registry)
- Never committed to git (`.gitignore` excludes `secrets/`)
- Gateway reads secrets, Nanobot receives them as environment variables

## Container Security

- **Resource limits**: 1 CPU, 1GB RAM for Nanobot
- **Workspace restriction**: `restrictToWorkspace: true` prevents file access outside `/app/workspace`
- **Read-only secrets mount**: Gateway mounts `secrets/` as `:ro`

## HTTPS

- Gateway should be behind Cloudflare Tunnel, nginx, or similar reverse proxy with TLS
- Admin Panel on Vercel has HTTPS by default
- Never expose the gateway over plain HTTP in production

## Recommendations

1. Use a strong, unique auth token (auto-generated is 64 hex chars = 256 bits)
2. Set up Cloudflare Tunnel for gateway exposure
3. Regularly rotate the auth token (`make rotate-token`)
4. Monitor audit logs for unexpected access patterns
5. Keep Docker images updated
6. Backup the Obsidian vault regularly (`make backup`)
